### 部门 KPI 背景

部门年度 KPI 是「提升工作流中的 AI 与自动化覆盖率」（具体数字不公开）。

我的实习岗位在招聘时就已经被明确定位为「AI 落地角色」 — **不是招我来打杂的，是招我来识别和解决可自动化的浪费**。

### 我在组里的位置

#### 不是「组长让我做工具」

没有人给我一份「待优化清单」。所有的 8 个小工具都是**我在执行日常运营工作时自己识别出来的**。

> 具体方法论见 [[认知#第三章方法论识别浪费到分发]]。

#### 工作模式

日常工作日里有大约 40% 时间在执行常规运营任务（群管理、数据审核、BI 看板查看等），60% 时间在识别任务中的浪费、设计工具、开发、测试、写文档、收集反馈迭代。

周末和晚上会用来处理比较重的开发任务（比如调研数据清洗与流转 Skill 的重开发任务）。

#### 组内的关系

- **Mentor**：客户体验组的正职运营，知道我在做工具，**技术实现全由我独立完成**。
- **同组正职**：是我工具的主要使用方。每个工具交付后我会主动跟进使用效果与迭代需求。
- **其他实习生**：组里有其他实习生，但我走的是独立的 AI 落地路线，没有重叠。

#### 话语权是怎么建立的

入职初期我被安排过写 SQL 的任务，但发现并不擅长。Mentor **主动**向他的 leader 请示，把 SQL 任务转给其他同学，让我继续 focus 到 AI 提效方面。

但「具体怎么提、提哪些、进程规划」leader 和 mentor 都没说 —— 给的是一个高度自主的空白区。6 个工具的方向、优先级、节奏，**全部是我自己识别出来的**，不是 mentor 给的待办列表。

**话语权的核心来源是「自己识别 + 自己交付 + 同事真在用」**，不是被授权。

### 三类工具总览

> 详细技术细节、决策过程、关键卡点见「第二章 · 工具详档」。

8 个小工具不是孤立存在的，它们按解决的问题类型自然分为三类：

#### 类别一 · 本地数据清洗与流转（2 项独立工具）

处理 Excel 数据清洗、跨平台数据流转等**不涉及浏览器 UI 操作**的任务。

| 工具 | 频率 | 覆盖 | 交互方式 | 状态 |
|------|------|------|---------|------|
| 任务整理脚本 | 每日多次 | 1 项实习生工作 | 脚本运行 → 输出文件 | 已交付 |
| 调研数据清洗与流转 | 月度 | 1 项**正职**工作 | Skill | 已交付 |

**关键词**：业务规则显式化 · 19 步跨平台流转 · 半自动化 + 人工节点审核。

#### 类别二 · 浏览器自动化（1 项独立工具集合）

替代所有需要人工频繁与浏览器交互的重复操作。

| 工具 | 频率 | 覆盖 | 交互方式 | 状态 |
|------|------|------|---------|------|
| Playwright 浏览器自动化工具箱 | 每日 / 每周 | **5 项**实习生工作 | WebUI 配置面板 | 已交付（2026.05 由 Chrome 扩展重构升级）|

**关键词**：Playwright 编排 · WebUI 配置面板 · 批量操作 · 断点续传。

> 该工具的前身是一个 Chrome 扩展（MV3），2026 年 5 月被重构为一个 Playwright 工具箱集合，统一替代所有浏览器内重复交互操作。当前覆盖的 5 项工作包括 2 项周度爬取（安装进度批量跟进、供应链安装跟进）与 3 项每日自动化（客户培训通知、群消息推送 / 供应链群消息维护、私域群消息管理）。

#### 类别三 · 内部系统数据提取（3 项工具）

程序化地从内部系统拉取数据——API 调用、图片预爬、浏览器端数据提取。

| 工具 | 频率 | 覆盖 | 交互方式 | 状态 |
|------|------|------|---------|------|
| BI 看板定时导出 | 每日 | 1 项实习生工作 | Skill（对话式）| 已交付，后台无人值守取数（已接 3/8 看板，正在扩展）|
| 图片审核 WebUI | 每日 | 1 项实习生工作 | WebUI | 已交付 |
| 商户系统数据自动化 | 周度 / 每日 | 1 项已交付（信息批量获取）+ 1 项已交付（工单批量下发）| WebUI + 人工最后闸门 | 已交付 |

**关键词**：API 鉴权逆向 · 并发预爬 · Headed 模式 · Chrome DevTools。

#### 三个类别的统一模式

所有需要和人交互的工具，一律采用 **WebUI**（而非命令行或后台脚本），原因：同事不需要学技术，打开浏览器就能用。唯一的例外是 BI 看板导出——它被做成了内部 AI 应用平台的 Skill，同事通过对话式交互触发导出。

Chrome DevTools 是这 8 个小工具开发中**使用频率最高的单一工具**——CLI 可以走 agent，但我偏好手动挡直接看 Network / Elements / Application 面板。全部不依赖 AI。

#### 汇总

**8 个小工具，覆盖 12 项核心日常工作中的 10 项（83%），外加 2 项正职月度任务（调研数据清洗与流转、数据复核）。** 单项提效区间 4× ~ 35×。

未自动化的 2 项均受外部约束不可行（IM 客户端风控）（详见第三章）。

**整体工时影响（每日）**：
- 周一/周二/周三/周日：≈7.5–8 h → ≈3 h 10 min（节省 ≈4.5 h，提效约 56–60%）
- 周四/周五/周六：≈6.5 h → ≈3 h（节省 ≈3.5 h，提效约 54%）

### 两个止损与边界探索案例（概述）

不是所有识别出来的浪费都该做、都能做。

1. **内部 ROI 不成立**：可视化 Excel 流水线项目——投入数轮迭代后判断方案复杂度超出收益，主动止损。
2. **外部约束不可突破**：内部 IM 客户端群管理自动化——建群操作（约 1 h/天）+ 退群操作（约 1.5 h/天）合计每日约 2.5 h 工作量，4 条技术路径全部被平台级风控机制堵死。

细节见「第三章 · 止损与边界探索」。

### 我离开时（或现在）组里什么情况

#### 如果我明天离开

- 已交付的工具继续运行。BI 看板导出是全自动的，不需要人维护；Playwright 工具箱、图片审核 WebUI、任务整理脚本、信息批量获取等均已交付给同组使用方。
- 但**工具的维护和扩展会成为问题**。文档写得足够清楚，但如果新需求出现（比如 BI 平台改版、内部系统页面结构变化），组里没有人有能力和时间接手修改——因为这些工具是 AI 协作生成的，维护同样需要 AI 协作能力和业务判断力，两者组里都缺。
- 这揭示了部门 AI 自动化 KPI 的第三个卡点（见 [[认知#第五章观点ai-红利期]] 第六章）。

#### 调研数据清洗与流转的当前状态

- **已交付**，打包为内部 Agentic Coding 平台的 Skill 标准包。
- 覆盖 **19 步**跨平台数据流转，含 WebUI 实时可视化进度展示。
- 从数据提取、清洗、校验到流转分发全部由 Skill 自动完成，SOP 内置在 Skill 中。
- **半自动化设计**：关键节点保留人工确认，同事在对话式交互中推进流程，随时可查看当前步骤状态。
- 正职同事通过对话式交互一键触发，无需学习命令行或操作复杂界面。

#### 数据复核的当前状态

- **已交付**，正职月度任务（月底执行），同样打包为 Skill 形态。
- **4 大模块分步自动化**：数据获取 → 规则匹配 → 复核判定 → 结果输出。
- 含 WebUI 实时可视化，同事可随时查看当前模块进度。
- 半自动化设计，关键判定节点保留人工审核。

### 对这段经历的 3 个核心认知

#### 认知 1：我不是「写代码的人」，是「自己发现问题、定义需求、用 AI 交付」的角色

招聘时岗位描述里没有写「开发工具」。所有工具都是我在执行日常任务时自己识别出来的。

更重要的是——**我不会写代码。** 所有硬技术实现来自 LLM。我做的事是：

- 识别出「这件事不该人做」。
- 把业务需求翻译成 LLM 能理解的精确描述。
- 在 LLM 给的方案中做选择。
- 用 Chrome DevTools 验证 LLM 的输出。
- 设计用户怎么用（WebUI、README、分发）。

**这说明我的价值不在代码层面，在决策层面。**

#### 认知 2：技术实现不是最难的，识别「什么值得做」和「怎么做合适」才是

8 个小工具的技术含量都不高——任何熟练使用 AI Coding 工具的人都能在数小时内生成类似的代码。但每个工具背后都有一个判断：

- 为什么 BI 那个项目用 API 而非浏览器自动化？
- 为什么浏览器自动化被统一成一个 Playwright 工具箱而非继续做独立扩展？
- 为什么商户系统数据自动化不做成全自动而留最后一道人工闸？
- 为什么调研数据清洗与流转在 Skill 中保留人工确认节点？
- 为什么内部 IM 群管理自动化 4 条技术路径全碰壁后选择了放弃？

**这些判断才是我的价值所在。** AI 能实现每一条技术路径，但选对路、知道什么时候该放弃，是我的工作。没有人替我判断。

#### 认知 3：分发和采纳是工具价值的最后一环

工具做完了不等于完成。完成 = 同事愿意装、装了能跑、跑出问题能定位。为此我做的是：

- 每个工具都有 README，5 分钟内能上手。
- 所有需要和人交互的工具一律用 WebUI——打开浏览器就能用，不需要学命令行。
- BI 导出是唯一的例外：打包成内部 AI 应用平台的 Skill，同事通过对话式交互触发。
- 任务整理脚本提供配置化的 YAML 规则文件，改规则不用碰代码。
- Playwright 工具箱提供 WebUI 配置面板，选任务、调参数、一键运行。

**如果你做的工具只有你能用，那就只是一段脚本，不是产品。**

### 正职月度任务交付 · 关键事实

#### 调研数据清洗与流转为什么采用 Skill 形态

- 19 步跨平台数据流转，涉及多个内部系统，鉴权方式各不相同。
- 正职同事的日常交互习惯是与内部 Agentic Coding 平台对话，因此最终打包为 Skill 标准包，而非独立 WebUI。
- Skill 内置完整 SOP、执行脚本和可视化前端，同事通过对话式交互推进流程，关键节点保留人工确认。
- 半自动化设计：人在过程中明确知道「机器刚才做了什么，我接下来要确认什么」，确保同事敢用、用错了找得到原因。

#### 数据复核的设计思路

- 月底一次性任务，4 大模块（数据获取 → 规则匹配 → 复核判定 → 结果输出）。
- 同样采用 Skill 形态，贴合正职同事的对话式工作习惯。
- 分步自动化 + WebUI 可视化进度，关键判定节点保留人工审核，避免全自动下的责任归属不清。

#### 最想优化哪个已交付工具

- 任务整理脚本。当前规则在 YAML 里，改规则不用碰代码，但每次改完需要跑回归测试。
- 理想状态：规则配置 WebUI 小界面，勾选条件 → 预览结果 → 确认执行。让维护门槛从「会改 YAML」降到「会用浏览器」。

#### 哪个工具对业务影响最大

- **节省工时**：Playwright 工具箱最大（覆盖 5 项工作，常态每天累计省 1 h+，周度峰值省 2 h+）。
- **全自动化无人值守**：BI 看板定时导出（每天省 30 min，已接 3/8 看板，扩展后还会增加）。
- **改变工作流形状**：Playwright 工具箱影响最大（把多个平台的重复浏览器操作统一成一个配置面板，同事不再被绑死在固定时间段）。
- **判断力展示**：调研数据清洗与流转最大（Skill 中保留人工确认节点的设计，证明了不是「能做就全做」）。

### 可迁移能力

- **AI 协作开发**：熟练使用 Claude Code、Codex 等 Coding Agent 与内部 Agentic Coding 平台 Skill 体系，完成需求拆解、代码生成、调试、文档撰写；能独立判断 AI 输出可用性。
- **自动化工具开发**：Python 脚本、Chrome Extension（MV3）、Playwright 工具箱、FastAPI + React 全栈 WebUI、定时任务、抓包逆向、HTTP API 适配。
- **产品思维**：从用户故事出发设计工具，关注分发与可用性（WebUI 配置面板、桌面快捷方式、Skill 标准包）。
- **业务理解**：餐饮 SaaS 客户体验侧实际业务，能将业务痛点准确翻译为技术需求。
- **工程意识**：最小可行链路验证、关键上下文文件化、生成与评估分离、主动叫停低收益方案、识别外部约束边界。

## 第二章 · 工具详档

> 数字分身被追问「讲讲 BI 自动化那个工具你是怎么做的」「浏览器自动化碰到的最大卡点是什么」时调用本章。
>
> 8 个小工具（统一为一个工具包）按解决的问题类型分为三个类别，外加 2 项正职月度任务详档。每个工具按固定 4 段组织：业务现场 / 判断瞬间 / 关键决策 / 分发与影响。
>
> 注意：本章所有真实工时数据、覆盖面、提效比例的单一真相源为第四章「附录：工作留痕底稿」，请勿在本章直接修改这些数字。

### 类别一 · 本地数据清洗与流转（2 项独立工具）

**类目概述**：处理不涉及浏览器 UI 操作的数据任务——Excel 清洗、规则匹配、跨平台数据流转。核心挑战不是技术实现，是**把隐性业务规则显式化**。

覆盖：1 项实习生工作（每日多次）+ 1 项正职工作（月度，已交付）。

---

#### 工具 1 · 任务整理脚本

##### 业务现场

每日要从大 Excel 中按规则筛选当日付费汇总对象、分类、分发。人工单次约 4 min 40 s，日均多次。**最大成本不是时间，是规则不稳定** — 不同人按不同侧重筛，产出标准漂移。优化后单次约 1 min（节省 80%），且规则稳定一致。

##### 判断瞬间

规则是隐性的，但可以显化。把同事脑里的规则逐条问出来、写成 if/else，人的工作从「执行规则」变成「维护规则」。这一步是**业务翻译**，不是技术开发。技术上 Python 30 行搞定——LLM 几秒就能生成——难的是把规则问清楚。

##### 关键决策

- **Excel vs Python + YAML**：Excel 模板维护成本高、跨人不可读。Python 脚本 + 配置化规则，规则写在 YAML，同事改规则不用碰代码。
- **规则漏洞迭代**：前两周持续暴露边界 case，加快迭代节奏，把「找规则漏洞」变成 SOP。
- **列名容错**：数据源 Excel 列名经常被业务方改，加一层列名别名映射，避免每改一次都报错。

##### 分发与影响

- 配套规则维护文档，明确「修规则前先 git diff、跑一遍历史数据回归」。
- 输出文件命名带日期，自动归档。
- 这个工具揭露了一个更大的问题：**很多重复劳动的本质是「业务规则不显式」**。AI 帮不上显式化规则的工作——它不在业务现场，不知道规则是什么。这是「业务一线 + 能动手」这个组合稀缺的证据。

**关键词**：业务规则显式化 · YAML 配置 · 列名容错 · git diff 回归。

#### 工具 2 · 调研数据清洗与流转

##### 业务现场

调研数据清洗与流转：数据要跨 **19 步**、跨多个内部平台流转，**每一步格式不同、责任主体不同**。原本由正职手工跑，单次约 2–3 小时，每月做一次。这是组里**唯一没被自动化覆盖的高耗时正职月度任务**。

##### 判断瞬间

直觉是「全自动化」 — 19 步一气呵成。但深入梳理之后，我**故意没这么做**。

19 步里有多个「跨平台上传 / 下载」节点 — 这些节点一旦机器失败，排查链路极长，且**责任归属不清**（机器替谁操作？谁背锅？）。决定架构：**半自动化 + 人工节点审核** — Skill 自动推进大部分步骤，关键节点停下来等人工确认，人在过程中明确知道「我刚才确认的是什么」。

##### 关键决策

- **内部 Agentic Coding 平台 Skill 形态**：正职同事的日常交互习惯是与内部 Agentic Coding 平台对话，因此最终打包为 Skill 标准包，而非独立 WebUI。同事通过对话式交互触发流程、查看进度、确认节点。
- **19 步 SOP 内置**：完整的数据提取、清洗、校验、流转分发流程写成 SOP 嵌入 Skill，LLM 根据 SOP 自主决策每一步该调用哪个脚本、处理哪个平台。
- **WebUI 实时可视化**：前端用于长程人机协同过程的可视化，同事可随时查看当前步骤状态（待 / 跑 / 成 / 败），不需要记步骤序号。
- **人工节点审核**：关键跨平台上传 / 下载节点保留人工确认，确保责任归属清晰。人在环内，机器不会擅自操作敏感数据。

##### 分发与影响

- 打包为内部 Agentic Coding 平台的 Skill 标准包，同事通过对话式交互一键触发。
- 正职同事无需学习命令行或操作复杂界面，对话即可推进 19 步流程。
- 这是 8 个小工具里**流程最长、克制也最深**的一个。**它的存在不是为了证明我会做全自动化，是为了证明我会判断哪里不该全自动。**

**关键词**：19 步跨平台 · 半自动化 · 人工节点审核 · 责任归属 · 内部 Agentic Coding 平台 Skill · SOP 自动化 · WebUI 可视化。

#### 工具 3 · Playwright 浏览器自动化工具箱

##### 业务现场

组里多个内部平台没有批量操作功能。运营同事每天/每周要在这些平台上做逐条点击、逐项勾选、逐页翻查。这不是某一个平台的问题，是**多个平台的共同特征——平台缺功能，但没人会为此立项**。

最初识别出的最高频场景是某周度的群标签批量打标任务：当时整段时间约 60 min，眼睛和手都被锁死（这是 Chrome 扩展时代的版本）。

##### 判断瞬间

2026 年 3 月进组时，我先做了一个 Chrome 扩展（MV3）专门解决群标签平台的批量操作。两个月后回头看，发现组里还有 4 个其他重复浏览器交互的场景——分别是周度的两类安装跟进爬取，和每日的客户培训通知 / 群消息推送 / 私域群消息管理。

**单独为每个平台写一个扩展，维护成本太高。** 2026 年 5 月，我决定把 Chrome 扩展重构为一个 Playwright 浏览器自动化工具箱——统一架构，覆盖所有需要浏览器交互的重复操作。LLM 生成单个脚本，我设计「配置面板 + 脚本编排 + 失败记录」的集合框架。

重构完成后实测：周度爬取任务从 ≈1 h 压到 ≈7 min（节省 89%）；每日操作类任务（客户培训通知 / 群消息推送 / 退群消息）从 30+ min 压到 1 min 内、可无人值守。

##### 关键决策

- **从 Chrome Extension MV3 到 Playwright 工具箱**：MV3 扩展受限于浏览器沙箱、跨标签状态维护复杂、权限模型有限。Playwright 无沙箱限制，能操控浏览器所有行为，且脚本可复用、可编排。
- **WebUI 配置面板**：选任务、调参数（批量大小、单次间隔、断点续传），点「运行」后 Playwright 启动浏览器自动执行。谨慎的同事可以小批跑，激进的可以一把梭。
- **分块执行 + 进度续传**：大量任务一次性灌入会爆子进程。任务切成小批次写入本地存储，每批执行完更新游标，异常重启从断点继续。
- **DOM 解析**：内部平台用了虚拟列表，直接 query 拿不到全部节点，需要触发滚动 + 监听节点挂载。这部分信息在 Chrome DevTools Elements 面板里观察获取，喂给 LLM 生成精确选择器。
- **失败落地**：失败任务自动输出到 .csv，方便手工兜底。

##### 分发与影响

- WebUI 面板一键启动，3 步上手：选任务 → 调参数 → 点运行。
- 统一了 5 个不同重复浏览器交互场景，从「5 个独立痛点」变成「1 个工具箱」。
- 最大的隐性改变：同事不再被绑死在固定时间段坐在电脑前操作——跑起来后可以做其他事，回来检查失败记录即可。

**关键词**：Playwright 编排 · WebUI 配置面板 · 从 MV3 到工具箱 · 分块续传 · Chrome DevTools · DOM 虚拟列表。

#### 工具 4 · BI 看板定时导出

##### 业务现场

内部 BI 平台日常需要查看 8 张看板的数据。原本要手工登录平台、逐张点击导出 Excel，单次约 30 min，每周 5 天，是组里**最稳态的浪费**之一。当前已接入自动取数 3/8 看板，剩余 5 张正在扩展中。

##### 判断瞬间

第一反应是写脚本调用 BI 平台的导出 API。但平台没有对外文档，鉴权机制不是简单的 token。这不是「能不能做」的问题，是「值不值得花 2 天开发换每天省 30 min」的问题。计算 2 周的回本周期，结论值得。但关键判断还在后面：浏览器自动化（模拟点击）简单但稳定性差，API 逆向难度高但一次解决永久收益。**选 B — 做杠杆，不做苦力。**

##### 关键决策

- **鉴权逆向（Chrome DevTools 核心场景）**：Cookie + 10 余项自定义 HTTP Headers。LLM 给的初版方案只复刻常见 Headers（Content-Type、Accept、Referer），实际请求被平台拦截。自己在 Chrome DevTools Network 面板里逐条对比真实请求 vs LLM 给的列表，补齐了业务自定义 Headers（其中几个 key 是缩写，AI 无法推理出语义）。这个「观察 → 假设 → 验证」循环重复了多轮。
- **同步 / 异步双版本**：某些大表导出走异步，需要轮询任务状态。两版分别封装，调用方无感知。
- **定时任务**：当前已配置 3 套独立定时任务（覆盖 3/8 看板），每张看板可单独开关、单独配置参数。架构上为剩余 5 张看板预留接入位，扩展时只需补 Headers 和导出 endpoint。
- **打包为 Skill（而非 WebUI）**：这是 8 个小工具中唯一不采用 WebUI 的。做成了内部 Agentic Coding 平台的 Skill 标准包，同事通过对话式交互触发——在聊天框里说「导出今天的看板」即可。

##### 分发与影响

- 配套 README 标注：出错时看哪个日志、Cookie 过期怎么续。
- 已接 3/8 看板，无人值守取数，剩余 5 张按相同架构扩展（补 Headers 和导出 endpoint 即可）。
- 最大的隐性改变：把「打开浏览器登录导出」这种需要人坐在电脑前的同步动作，变成「在聊天框说一句话或定时触发」的异步动作——同事的注意力不再被取数环节占用。

**关键词**：鉴权逆向 · Chrome DevTools Network 抓 Headers · 同步异步双版本 · Skill 标准包 · 对话式交互。

#### 工具 5 · 图片审核 WebUI

##### 业务现场

服务卡审核场景：每日数百张图片来自外部平台，内部审核侧没有批量预览界面，**每张要单独点开等加载 2–5 秒**。单次审核 1–1.5 h，80% 时间在等加载，不是在判断。优化后单次约 20 min（节省 78%），人只做判断、不再等加载。

##### 判断瞬间

浪费的本质是**串行 IO 阻塞**。判断很简单 — 把加载和审核解耦，人就只做判断。不需要 AI 判图能力，只需要把图片预先拉到本地，然后让人瀑布式地看。

##### 关键决策

- **方案**：Python 后端并发预爬 + WebUI 前端瀑布流一次性渲染。LLM 生成代码骨架，我设计交互体验。
- **Referer 防盗链**：部分图片来源做了 referer 限制，Python 侧请求要伪造 referer。这个信息在 Chrome DevTools Network 面板里确认。
- **瀑布流布局**：图片高度未知时用 onload 回调动态计算高度，避免布局抖动。
- **审核结果回传 WebUI**：每张图旁加「通过 / 驳回」按钮，异步 POST 到后端汇总。交互设计的目标是让审核员从「等加载」变成「只看图、只判断」。

##### 分发与影响

- 打成单个可执行文件 + 启动脚本，双击运行、浏览器访问 localhost。
- **不依赖任何后端服务，数据不出本机**，不存在合规审批延迟。
- 配套文档明确：这是「一次性会话工具」，审核完即关，不留持久化。

**关键词**：预爬 · 瀑布流 · Referer 伪造 · 本地运行无审批 · WebUI。

#### 工具 6 · 商户系统数据自动化

> 该工具下挂两个子能力：（A）信息批量获取——已交付，按周一/周三既定节奏在用；（B）工单批量下发——已交付，每日在用。两者共享底层架构（Playwright + Headed + WebUI 配置面板），只是上层业务流程不同。

##### 业务现场

**子能力 A（已交付）· 信息批量获取**：周一/周三需要从内部外呼系统批量提取一组目标客户信息。人工逐条复制粘贴单次约 40 min；优化后单次约 6 min（节省 75%）。这套抓取逻辑可复用到调研类场景（调研类名单采集）。

**子能力 B（已交付）· 工单批量下发**：每日要从内部团购工单平台批量拉取待处理工单，按上传模板格式注入到另一个系统下发。人工逐条复制粘贴约 30 min/天，链路上需要频繁「本地 ↔ 网页」上传下载交互，跨表复制粘贴 7 次流转填充模板。优化后鼠标点击仅 5 次即可完成两个板块，30 min → 3 min（节省 89%）。

##### 判断瞬间

这是前面几个工具最自然的延伸 — 鉴权 + 浏览器自动化 + 数据格式化，套路已熟。但真正的判断是：**做不做全自动化？**

答案是「半自动化」 — 数据提取自动，模板生成自动，**但最后一步「点上传」留给人**。原因：全自动化下如果一次错误的批量上传写入了错误数据，**事后归责和回滚极困难**。半自动化让人在最后一秒留一道闸。

子能力 B 的实现路径：最初判断需等待平台开放 API，但后来通过**同事协助获得内部接口支持**（不走公开 API 路径），结合 Playwright 完成浏览器端自动化，最终走通了混合实现方案。这是「卡住的时候找资源而不是死等」的典型案例。

##### 关键决策

- **Playwright + Headed 模式**：数据 prepared 之后，浏览器停在「上传按钮前」，人确认后再点。Headed 让人能看到机器到底做了什么——不是黑箱操作，出错也好定位。
- **商户系统列表分页 + 懒加载**：scroll 到底才能拿全数据。Playwright 的 waitForLoadState 不够，要监听 network idle + DOM 节点数稳定。这部分行为特征在 Chrome DevTools 里观察后喂给 LLM。
- **上传模板约束**：字段顺序、空行规则、必填项约束，只有跑过一次失败才知道 — 这部分是**和业务方对齐**而不是技术问题。
- **WebUI 配置面板**：启动后弹本地 WebUI，选时间区间、选数据范围、选输出路径。一键启动，机器跑完后主动暂停并提示「请检查 XX 字段后手动点上传」。
- **同事协作**：商户系统工单部分的自动查询负责同事、自动上传负责同事等功能，依赖内部同事提供的接口支持，不是纯逆向/Playwright 能独立完成的。

##### 分发与影响

- WebUI 交互，不需要学命令行。
- 工单批量下发两个板块所需的模板已全部内置，包含自动查询负责同事、自动上传负责同事、不依赖内部 IM 客户端即可自动导出最终明细。
- **这个工具是我 8 个小工具里最满意的 — 不是因为技术最重，是因为我克制了「能做就全做」的冲动，并且在卡住时找到了协作资源。**

**关键词**：半自动化 · 责任边界 · Headed 模式 · 人工最后闸门 · WebUI · Chrome DevTools · 同事协作。

#### 工具 8 · 数据复核（Skill 形态）

##### 业务现场

月底数据复核是正职月度任务，涉及数据获取、规则匹配、复核判定、结果输出 **4 大模块**。原本由正职手工逐条核对，耗时且容易遗漏。任务性质是**规则明确但步骤多、跨系统流转频繁**。

##### 判断瞬间

和调研数据清洗与流转一样，第一反应是做成全自动。但复核工作的核心是**判定**——哪些该通过、哪些该驳回，这个判定需要人对业务上下文的理解，机器无法替代。因此选择 **Skill 形态 + 分步自动化**：机器处理数据获取和规则匹配，人在关键判定节点做最终决定。

##### 关键决策

- **内部 Agentic Coding 平台 Skill 形态**：贴合正职同事的对话式工作习惯，通过对话触发流程、查看进度、确认节点。
- **4 大模块分步自动化**：数据获取 → 规则匹配 → 复核判定 → 结果输出，每一步独立可验证，出错可定位到具体模块。
- **WebUI 实时可视化**：同事可随时查看当前模块进度，不用记步骤序号。
- **人工审核节点**：复核判定模块保留人工确认，机器提供数据支撑和规则匹配结果，最终判定权留给正职同事。

##### 分发与影响

- 打包为 Skill 标准包，月底正职同事通过对话式交互一键触发。
- 将原本分散在多个系统的复核流程统一到 Skill 中，减少切换成本。
- 与调研数据清洗与流转采用同一套 Skill 交付模式，降低正职同事的学习成本。

**关键词**：月底任务 · 4 大模块 · Skill 形态 · 分步自动化 · 人工判定节点 · WebUI 可视化。

## 第三章 · 止损与边界探索

> 数字分身被问「你有没有主动停掉过什么项目」「有没有做砸过的事」「有没有想做但做不成的」时调用本章。
>
> 这里记录了两种不同类型的边界判断：**案例 A 是内部 ROI 不成立**（能做但判断不该做），**案例 B 是外部约束不可突破**（想做出发就被堵死）。两个案例的价值不在「做成了什么」，在「知道什么时候该停、什么时候该认」——这是比工具数量更稀缺的判断力。

### 案例 A · 可视化 Excel 流水线（内部 ROI 不成立）

#### 项目背景

实习期间接到一个需求：把若干份运营 Excel 做成「可视化流水线看板」，实时反映关键指标。

项目目标是做一个 ComfyUI 风格的 Excel 处理工具——非技术用户通过拖拽节点定义数据处理流程，本质上是一个可视化的 Power Query 替代品。经历了**两代架构，6 轮 Sprint 迭代**。

#### 投入与产出预估（动手前）

- 预估开发 2–3 天。
- 持续维护成本预估极低。
- 日均价值预估为 10+ min 节省。
- **看起来 ROI 不错。**

#### 最小链路验证（做了第一版 demo 之后）

按我的方法论，先跑最丑的 MVP，当天给使用方看。

实际跑下来发现 ROI 并不成立——使用方对「打开 Excel 看原始数据」的习惯路径黏性远高于预期，看板带来的提效集中在「扫描感」而不是真实时间节省；同时 Excel 列名/口径调整频次不低，看板的隐性维护成本被低估。

#### 核心问题诊断

##### 问题 1：没有节省真正的「时间」，只节省了「感觉」

使用方真正想要的是「更爽的查看体验」，但原有 Excel 流程在时间维度上已经足够快，看板没有解决一个真实的阻塞点，只是把它包装得更漂亮。

##### 问题 2：看板的维护成本被低估了

Excel 的列名、字段口径、指标定义都在持续微调。每调整一次，看板就需要同步调整数据源映射。维护成本不是 0，而是「不定期出现但累积起来不低」的隐性成本。

##### 问题 3：使用方并没有"痛苦"到愿意改变习惯

看板的 ROI 公式应该是：**（节省的时间 - 维护成本）/ 学习成本**。在这个项目上，分子趋近于 0，分母不为 0 — 负数。

#### 决策：主动叫停

没有要求使用方迁移，没有继续迭代，**直接停掉**。

**停止原因**：经历了两代架构、6 轮 Sprint 迭代之后，判断节点式可视化数据处理涉及复杂的状态管理、节点间依赖解析、实时数据预览和边界情况处理，短时间内无法通过单人 AI Coding 达到可用水准。这个项目触碰到了当前模式的能力边界——不是所有问题都适合用「快速原型 → 迭代」的方式解决，有些问题的复杂度需要专业团队和长周期投入。

**教训**：触碰到边界的时候要及时停止。继续投入不会突破边界，只会浪费时间。正确的做法是退回来，进一步清晰化业务优化方向——先搞清楚运营同事真正需要的是什么层级的 Excel 处理能力，再决定用什么方式实现，而不是执着于「可视化节点」这个形态。

留下一份简短的复盘，标注后续遇到类似需求时的 3 个前置判断条件：数据源口径短期内是否稳定、原有手动流程单次是否真的耗时高、使用者是否真的愿意切换工作流。

#### 这件事的真正价值

##### 对我个人的校准

8 个小工具能跑出来，**前提是第 7 个工具没跑完**。如果我把这个项目硬推上线：

- 会占据 2–3 天的开发时间。
- 上线后会消耗不定期的维护时间。
- 整体 ROI 反而下降。
- 更隐蔽的代价：**我可能会因为这个项目被贴上「只会做看板」的标签**，损失在 AI 自动化方向上的定位清晰度。

##### 对方法论的反向校准

我的方法论是「识别浪费 → 业务翻译 → 最小链路验证 → 工具分发」。这个项目的失败是对**第一步「识别浪费」**的校准 — 它告诉我：

- 不是「重复 = 浪费」，是「重复 + 阻塞 + 隐性规则缺失 = 浪费」。
- 不是「用户提了 = 需求"，是「用户提的 + 最小链路验证通过 = 需求」。
- **最小链路验证的真正目的不是验证"能不能做"，是验证"做了之后用户会不会真的用"。**

##### 对 AI 时代的岗位价值的判断

AI 时代「能做」的项目变多了，"该做"的判断更稀缺。这个项目的意义在于：

- 技术能力足够做的东西，不等于组织能力需要的东西。
- AI 协作工程师最贵的肌肉，不是"写出多少代码"，是"在 2 小时内判断一个项目该做不该做"。
- **能停掉一个项目的判断力，和能启动一个项目的执行力，价值相当。**

### 案例 B · 内部 IM 客户端群管理自动化（外部约束不可突破）

#### 业务现场

组里有大量通过内部 IM 客户端（即时通讯工具）完成的建群和退群操作。**建群操作约 1 h/天 + 退群操作约 1.5 h/天，合计每日约 2.5 h**——建群、拉人、退群、转让群主、退群消息发送——全部手工在客户端里逐群操作。

从「识别浪费」的三个信号看，这个任务同时触发了**重复**（每天大量逐群手工动作）、**等待**（客户端操作有响应延迟）、**隐性规则**（哪些群该拉、哪些该退，部分依赖经验判断）。是理论上最该被自动化的浪费。

#### 技术尝试：4 条路径全部碰壁

决定尝试自动化。以下 4 条技术路径全部来自 LLM 的建议，每条在理论上都有成功案例，但在内部 IM 客户端上全部被堵死：

##### 路径 1 · UIA 控件树探测

LLM 建议通过操作系统级的 UI Automation 接口，直接读取客户端窗口的控件树，定位按钮并触发点击。

**碰壁原因**：该 IM 客户端没有暴露标准的 UIA 控件树。界面元素使用自绘引擎渲染，操作系统级别的 API 无法识别按钮和输入框的边界。

##### 路径 2 · 坐标录制回放

LLM 建议录制一次手工操作的鼠标坐标序列，然后脚本按坐标回放。

**碰壁原因**：客户端窗口内的元素位置不是固定的——群列表的滚动位置、窗口大小变化、新消息通知弹出——都会导致坐标偏移。录制回放 5 次，5 次的点击位置都不一样。

##### 路径 3 · 键盘导航替代

LLM 建议绕过鼠标坐标，用 Tab / 方向键 / 快捷键在客户端内导航焦点，模拟键盘操作。

**碰壁原因**：客户端的焦点模型极其复杂，Tab 顺序在群聊列表、聊天区域、侧边栏之间不可预测。且部分关键操作（群管理菜单）没有键盘快捷键，只能鼠标触发。

##### 路径 4 · CDP 远程调试注入

CDP（Chrome DevTools Protocol）是控制 Chromium 内核的标准协议。LLM 建议如果客户端内嵌了 Web 视图，可以通过 CDP 远程调试端口注入操控逻辑。

**碰壁原因**：该 IM 客户端的核心界面**不是 Web 视图**，是原生渲染引擎。CDP 完全不可用——没有调试端口、没有 WebView 实例。

#### 最终判断：必须依赖人工

4 条路径全部不可行之后，结论很明确：**这个 IM 客户端在设计层面就不支持自动化操作。** 无论技术方案多精巧，客户端的自绘渲染、风控策略与缺乏官方 agent 接口，共同构成了一道平台级的安全墙。**这类任务必须依赖人工，等待客户端官方放开 agent 接口才有重新评估的可能。**

#### 核心教训

这个案例的价值不在技术层面，在对 AI 能力边界的认知校准：

1. **LLM 给的技术方案在理论上都成立。** UIA 控件树、坐标回放、键盘导航、CDP 注入——这些在标准 Windows 应用或 Web 应用上都是成熟方案。LLM 没有骗我。

2. **但 LLM 不知道这个平台有风控。** 它不在业务现场，不知道这个 IM 客户端的自绘引擎、不知道风控规则的触发条件、不知道这个平台的设计目标本身就包含「防止自动化」。这些信息不在 LLM 的训练数据里。

3. **这不是技术判断的失败，是外部约束的绝对边界。** 和案例 A（Excel 流水线）不同——案例 A 是我判断「做了 ROI 为负」，案例 B 是「想做出发就被堵死」。两类边界需要两类判断力。

4. **某些系统的设计目标包含「阻止你做这件事」。** AI 无法突破设计层面的安全机制。知道什么时候该认输，和知道什么时候该坚持，同样重要。

#### 合规与脱敏说明

本案例中：
- 不出现 IM 产品真实名称（以「内部 IM 客户端」代指）。
- 不记录具体群名、群成员信息、业务场景细节。
- 工时取约数（「群管理 ≈1 h/天 + 退群 ≈1.5 h/天」），不写精确数字。
- 4 条技术路径的名称和原理是通用技术知识，不涉及内部信息。

| | 案例 A · Excel 流水线 | 案例 B · IM 群管理 |
|------|------|------|
| 停止原因 | 内部判断：ROI 为负 | 外部约束：平台风控 |

### 为什么这些案例不进简历

简历的角色是「展示做成的事」。止损和边界故事的角色是「展示判断力」。

**做成的事靠数字说话，判断力靠故事说话。**

数字适合放在简历里，故事适合在对话中展开。两者的边界不清，会让访客对你的印象混乱。

> **关键句**：「能做但不该做」是判断力。「想做但做不成，知道为什么做不成」也是判断力。两种判断力加起来，才是 AI 协作工程师最贵的肌肉。
